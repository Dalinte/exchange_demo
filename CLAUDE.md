# CLAUDE.md

Контекст проекта для AI-ассистентов. Этот файл автоматически загружается Claude Code.

## О проекте кратко

Демо-биржа криптовалют с paper trading. Анонимные пользователи получают виртуальные $10,000 USDT и торгуют против реальных рыночных данных Binance. Никаких реальных денег.

**Стек:**

- Next.js 16 (App Router)
- Tailwind
- shadcn/ui
- NestJS 11
- Prisma
- PostgreSQL 18
- Zod
- TypeScript везде
- Lightweight Charts
- Binance public WebSocket — единственный источник рыночных данных пока
- npm workspaces (монорепо)

**Статус:** активная разработка, фаза MVP.

## Структура репозитория

```
apps/api          NestJS backend — REST + WebSocket gateway, бизнес-логика, соединение с Binance
apps/web          Next.js frontend — App Router, Lightweight Charts, server + client components
packages/shared   Zod-схемы и TS-типы, шарятся между api и web (single source of truth)
```

При cross-cutting изменениях (например, добавил поле в order) обновляй сначала `packages/shared`, потом api, потом web. Никогда не дублируй типы между приложениями.

## Жёсткие правила

Это правила, которые я хочу видеть соблюдёнными без напоминаний.

### Деньги и числа

- Все денежные значения (цены, балансы, количества) в Prisma — `Decimal`, никогда `Float`.
- В коде для арифметики над деньгами используй `decimal.js`. Никаких `+`, `-`, `*`, `/` на сырых `number` для денег.
- Форматирование для отображения (округление для UI) — на самом краю (контроллеры, компоненты), не в бизнес-логике.

### Валидация и типы

- Каждый DTO, который пересекает сетевую границу (HTTP body, query, WebSocket message), — это Zod-схема в `packages/shared`.
- TypeScript-типы выводятся из Zod-схем через `z.infer`, не пишутся руками рядом со схемой.
- NestJS-эндпоинты валидируют вход через Zod pipe. Формы на фронте — через `zodResolver` из `@hookform/resolvers/zod`.
- Никаких `class-validator`, никакого `class-transformer`. Zod — единственная библиотека валидации.

### Целостность ордеров и балансов

- Размещение ордера, обновление баланса и создание записи о сделке — в одной Prisma-транзакции. Никаких частичных состояний.
- У баланса два поля: `free` и `locked`. Лимитные ордера переводят средства из `free` в `locked` при размещении и обратно (или в counter-asset) при исполнении/отмене. Даже если фича сейчас вроде бы не требует `locked` — поле должно быть консистентно.
- Никогда не "прочитал баланс → посчитал новый → записал". Используй атомарные update-операции

### Real-time данные

- API держит ровно одно соединение с Binance на тип стрима, мультиплексирует на всех клиентов. Никаких соединений per-request или per-user.
- Поток данных: Binance WS → сервис в API → broadcast подписанным клиентам через NestJS WebSocket gateway. Фронт никогда не подключается к Binance напрямую.
- Формат WebSocket-сообщений — discriminated union из Zod-схем в `packages/shared`. На приёме всегда `WSMessageSchema.parse()`.

### Рендеринг на фронте

- По умолчанию — React Server Components. `'use client'` ставится только когда компоненту нужны `useState`, `useEffect`, обработчики событий или браузерные API.
- График, форма ордера, виджет балансов, лента сделок — клиентские компоненты.
- Шеллы страниц, layouts, метаданные, initial data fetch — серверные компоненты.
- Не импортируй server-only код в клиентские компоненты. Через shared-пакет или API-вызов.

### UI-компоненты

- Базовые UI-примитивы (кнопка, popover, dialog, dropdown, input, label, table, tabs, tooltip и т.д.) — из `apps/web/src/shared/ui/` (shadcn-сгенерированные обёртки над `radix-ui`). Прежде чем писать свой компонент или CSS-класс — посмотри, есть ли уже подходящий в `shared/ui/`.
- Если нужного примитива нет — генерируй через `npx shadcn@latest add <name>` из `apps/web/`. Не пиши собственные обёртки над `radix-ui` и не плоди bespoke CSS-классы для типовых интерактивных паттернов.
- Semantic-токены shadcn (`--color-primary`, `--color-accent`, `--color-destructive`, `--color-popover`, `--color-border` и т.д.) уже замаппены на проектные `--bg-*` / `--text-*` / `--down` в `apps/web/src/app/styles/globals.css`. Поэтому `variant="outline"`, `variant="destructive"`, `bg-accent`, `bg-popover` и т.п. дают проектные цвета без `className`-оверрайдов. Если кажется, что нужен оверрайд — сначала проверь, нет ли подходящего варианта/размера у компонента.
- Композиция классов — через `cn()` из `@/shared/lib/utils` (clsx + tailwind-merge), не через конкатенацию строк.

### Состояние и data layer

- React Context **не используем**. Для глобального клиентского состояния — Zustand (по одному стору рядом со своими потребителями: `entities/<resource>/model/`, `widgets/<name>/model/`, `shared/ws/`), для server state — TanStack Query. Context даёт каскадные ре-рендеры по всему поддереву и плохо селективен.
- Серверное состояние (то, что приходит с API) живёт только в Query Client. В Zustand — только локальный UI-state и параметры, которые не имеют права на server-side fetch (выбранный таймфрейм, активный символ из URL, открытость панелей).
- Если активный символ или другой параметр читается из URL — синхронизируем URL → store в одном месте (sync-компонент рядом со store), не дублируем в каждом потребителе.

## Соглашения по коду

- TypeScript strict mode везде. Никакого `any`. Используй `unknown` и сужай тип.
- Никаких barrel-файлов в apps (только в `packages/shared`).
- Именование файлов: kebab-case везде, включая React-компоненты (`order-form.tsx`, `candle-chart.tsx`). Внутри компонент по-прежнему `export function OrderForm`, но файл — kebab-case.
- Никаких сокращений в именах переменных и функций. Пиши полные слова: `percent` вместо `pct`, `priceDecimal` вместо `priceD`, `applyPercent` вместо `applyPct`, `quantityDecimal` вместо `qtyD`. Исключения — общепринятые в индустрии аббревиатуры (`id`, `url`, `dto`, `qc` для QueryClient — но даже их избегай, если есть полное имя).
- Функции вместо классов где возможно на фронте. NestJS использует классы — это паттерн фреймворка, нормально.
- `async/await` вместо `.then()`. Никогда не смешивай в одной функции.
- Обработка ошибок: бросаем типизированные ошибки в источнике, ловим и транслируем на границе (контроллер, gateway, route handler).
- При добавлении ключевой фичи — короткая заметка в `docs/<feature>.md` (зачем, поток, ссылки на код, известные ограничения) и ссылка на неё из секции "Архитектура" в `README.md`.

## Что делать, если непонятно

- Если бизнес-логика не описана — **спроси перед тем, как реализовывать**. Не угадывай в том, что касается денег, ордеров и балансов.
- Если нужно добавить зависимость — **скажи какую и зачем** перед установкой.
- Если рефакторинг затрагивает больше трёх файлов — **сначала покажи план**, потом исполняй.
- Для изменений схемы — `prisma db push`, без миграций. Я один разработчик, история миграций сейчас не нужна.
- Заполнение справочников (торговые пары и т.п.) — через `npm run db:seed` (`apps/api/prisma/seed.ts`). Idempotent upsert, безопасно перезапускать.

## Коммиты

- Conventional Commits: `feat:`, `fix:`, `refactor:`, `chore:`, `docs:`, `test:`, `style:`.
- Subject в настоящем времени, нижний регистр, без точки: `feat: add limit order placement`.
- Один коммит — одно логическое изменение. Cross-workspace правки (shared + api + web) могут быть в одном коммите, если это одна логическая правка.
